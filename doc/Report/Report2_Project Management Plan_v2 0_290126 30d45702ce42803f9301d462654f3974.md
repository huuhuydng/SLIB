### 2.2.2 Code Quality Management

To achieve our technical project objectives and maintain a low defect density, we implement the following standards:

- **Unit Test:**
    - Description: Test individual functions or components to make sure they work correctly according to the expected logic before being integrated into the full system.
- **Code Convention:**
    - Description: Apply consistent coding standards such as naming rules, formatting, and code structure for both the Mobile Application and the Web Dashboard.
    - Tool: ESLint, Prettier
- **Code Review:**
    - Description: Conduct peer reviews on newly written code to identify logical issues, performance problems, and potential security risks early. HuyNH is responsible for overseeing the quality of the Back-End architecture.
- **Continuous Integration and Continuous Deployment (CI/CD):**
    - Description: Automate the process of building, testing, and deploying the system to ensure that new code changes are validated and deployed regularly and reliably.
    - Tool: GitHub, Docker
- **Version Control:**
    - Description: Manage and track all source code changes using a shared repository to support team collaboration and avoid code conflicts.
    - Tool: GitHub

### 6.2 Source Code Management

The system’s source code is managed via GitHub, providing a centralized and version-controlled environment for collaboration. To maintain high code quality and consistency across the SLIB ecosystem, the team follows a structured management workflow:

- **Repository Organization**: The project adopts a Monorepo approach, in which all system components are maintained within a single repository. The source code is organized into separate directories corresponding to different technology stacks, including mobile (Flutter), backend (Spring Boot), ai-service (FastAPI), and frontend(ReactJS). This structure ensures clear separation of concerns while simplifying integration and version control across the system.
- **Branching Strategy**: The team utilizes the GitFlow model to manage development stages:
- main: Stores the stable, production-ready code.
- develop: Acts as the primary integration branch for all features.
- feature/[task-name]: Dedicated branches for individual tasks (e.g., feature/user-auth, feature/seat-booking). These branches are merged into develop only after passing a code review.

> 
> 

***Figure 4. Git Flow***

- **Commit Standards:** Team members must follow the Conventional Commits specification (e.g., feat:, fix:, refactor:) to ensure a readable and traceable project history.
- **Code Review & Quality Control:** Direct commits to main and develop are restricted.
- All changes must be submitted via Pull Requests (PR). The Leader (HuyNH) reviews the code for logic, security, and adherence to the project’s UI Color Palette.
- Automated Integration (CI): Every PR triggers a GitHub Actions workflow to automatically run unit tests and build Docker images. This ensures that new code does not break the existing system and is compatible with the Dockerized environment.
- **Environment & Database Versioning:** The PostgreSQL database is managed as "Infrastructure as Code" using Docker Compose.
- All database schema changes and initialization scripts are stored in the repository to ensure that every team member works on an identical local database environment.
